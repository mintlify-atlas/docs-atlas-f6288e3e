## Introduction

Node providers are essential participants in the Internet Computer network. Unlike traditional blockchain "miners," IC node providers operate independent data centers that host the network's infrastructure using novel cryptographic consensus mechanisms that don't require traditional mining.

<Info>
The Internet Computer Protocol uses new and novel cryptography that does not require "mining" but does require independent node providers to run the network infrastructure.
</Info>

## Who Should Be a Node Provider

Node providers should have:

- **Infrastructure Expertise**: Experience managing data center operations, server hardware, and network infrastructure
- **Security Knowledge**: Understanding of secure systems administration and network security
- **Commitment**: Ability to maintain high availability and respond to issues promptly
- **Capital**: Resources to purchase and maintain server hardware meeting IC specifications

#### Key Management and Rotation

The IC implements automatic key rotation for enhanced security:

```rust
// From rs/orchestrator/src/registration.rs:48-50
const DELAY_COMPENSATION: f64 = 0.85;

// Key rotation is coordinated across subnet nodes
// Gamma (rotation frequency) compensates for potential delays
```

**Key Rotation Process:**

1. **Disabled**: Key rotation not enabled for this node
2. **TooRecent**: Last rotation was too recent, waiting
3. **Rotating**: Generating new keys
4. **Registering**: Submitting new keys to registry
5. **Registered**: New keys confirmed in registry
6. **Error**: Rotation failed (alerts triggered)

<Note>
Key rotation status is exposed via the `orchestrator_key_rotation_status` metric. Monitor this to ensure healthy key rotation.
</Note>

#### Alert Configuration

<Warning>
Configure alerts for critical errors. Any increase in `critical_errors` counters requires immediate investigation.
</Warning>

Recommended alerts:

- **Node Offline**: No metrics received for 5 minutes
- **Registry Desync**: Registry version not updating
- **Key Rotation Failure**: `orchestrator_key_rotation_status{status="Error"}` set
- **Upgrade Failures**: Consecutive upgrade check failures
- **Resource Exhaustion**: CPU/memory/disk approaching limits
- **Process Crashes**: Replica restart attempts increasing

### Maintenance Windows

#### Planned Maintenance

For routine maintenance:

1. **Notify Network**: Submit maintenance proposal via NNS (if required)
2. **Schedule Appropriately**: Coordinate with subnet schedules
3. **Minimize Impact**: Use rolling maintenance across replicas
4. **Monitor Recovery**: Verify node rejoins subnet successfully

#### Emergency Maintenance

For critical issues:

1. **Assess Impact**: Determine if immediate action is required
2. **Document**: Log all actions taken
3. **Communicate**: Notify via appropriate channels
4. **Post-Mortem**: Analyze root cause and prevent recurrence

## Node Upgrades

Node providers must facilitate smooth upgrades:

### Replica Version Upgrades

Replica upgrades are coordinated by the Orchestrator:

```rust
// From rs/orchestrator/src/upgrade.rs
// The Orchestrator:
// 1. Monitors registry for new replica versions
// 2. Downloads upgrade artifacts from release URLs
// 3. Verifies cryptographic hashes
// 4. Coordinates with subnet for CUP (Catch-Up Package)
// 5. Stops old replica
// 6. Starts new replica version
```

<Info>
Replica upgrades are automatic and coordinated via the NNS. No manual intervention is required for normal upgrades.
</Info>

**Upgrade Flow:**

<Steps>
  <Step title="Registry Update">
    NNS governance approves new replica version and updates registry
  </Step>

  <Step title="Orchestrator Detection">
    Orchestrator detects version mismatch:
    ```rust
    // From upgrade.rs
    if current_version != target_version {
        start_upgrade_process()
    }
    ```
  </Step>

  <Step title="Artifact Download">
    Downloads replica binaries from configured URLs with verification
  </Step>

  <Step title="Coordinated Upgrade">
    Uses Catch-Up Package for safe state transition
  </Step>

  <Step title="Version Verification">
    Confirms new version is running and healthy
  </Step>
</Steps>

### HostOS Upgrades

HostOS upgrades are managed separately:

```rust
// From rs/orchestrator/src/hostos_upgrade.rs
pub async fn upgrade_loop(
    &mut self,
    cancellation_token: CancellationToken,
    mut backoff: ExponentialBackoff,
    liveness_timeout: Duration,
) {
    // Continuously checks for HostOS updates
    // Downloads from release_package_urls
    // Verifies SHA256 hash
    // Triggers HostOS reboot after installation
}
```

**HostOS Upgrade Process:**

1. Orchestrator detects new HostOS version in registry
2. Downloads upgrade package from configured URLs
3. Verifies package integrity (SHA256)
4. Requests HostOS upgrade via utility command
5. HostOS installs update and reboots
6. GuestOS VM restarts automatically after HostOS boot

<Note>
HostOS upgrades cause brief node downtime during reboot. The subnet continues operating with remaining nodes.
</Note>

## Security Best Practices

### Physical Security

- Restrict data center access to authorized personnel
- Use video surveillance in server areas
- Implement audit logs for physical access
- Secure spare hardware and backup media

### Network Security

#### Firewall Configuration

The Orchestrator manages firewall rules automatically:

```rust
// From rs/orchestrator/src/firewall.rs
// Firewall rules are synchronized from the registry
// Updated based on firewall_registry_version
```

**Required Ports:**

- `4100`: P2P consensus and transport (TCP/UDP)
- `8080`: HTTP ingress (for boundary nodes)
- `9090`: Replica metrics (internal)
- `9091`: Orchestrator metrics (internal)
- `9100`: Node exporter metrics (internal)

<Warning>
Do not manually modify firewall rules. The Orchestrator synchronizes rules from the registry to maintain network security policies.
</Warning>

### SSH Access Management

SSH access is controlled via the registry:

```rust
// From rs/orchestrator/src/ssh_access_manager.rs
// SSH keys are synchronized from the registry
// Metric: ssh_access_registry_version tracks last update
```

**SSH Access Policy:**

1. SSH keys are stored in the NNS registry
2. Orchestrator updates authorized_keys from registry
3. Access is granted based on registry permissions
4. Changes are auditable via registry history

<Info>
For development nodes, console access is available with username/password `root`/`root`. Production nodes disable console access.
</Info>

### Cryptographic Key Management

**Node Keys** (generated automatically):
- Node signing keys
- DKG (Distributed Key Generation) keys
- TLS certificates
- VET (Verifiable Encrypted Threshold) keys

**Storage:**
```
/var/lib/ic/crypto/
├── node_signing_pk.der
├── node_signing_sk.der
├── tls_cert.der
└── ... (other cryptographic material)
```

<Warning>
Never manually copy or modify files in `/var/lib/ic/crypto/`. Key management is handled by the replica's crypto component.
</Warning>

## Performance Optimization

### Storage Performance

IC nodes require fast storage for optimal performance:

- **Use NVMe SSDs** for state storage
- **Monitor FSTRIM**: Check `orchestrator_fstrim_duration_milliseconds`
- **Track Disk I/O**: Monitor via node_exporter metrics

```bash
# FSTRIM is run automatically via systemd timer
# Metrics location: guestos/fstrim/fstrim_tool.service
```

### Network Optimization

- Use low-latency network connections
- Monitor network metrics: `node_network_*`
- Ensure adequate bandwidth for P2P traffic
- Configure QoS if sharing network infrastructure

### Resource Allocation

**GuestOS VM Resources:**
- Allocate sufficient CPU cores (typically 8-32)
- Provide adequate RAM (32GB+)
- Ensure storage IOPS meet requirements

### Common Issues

**Node Not Joining Subnet**
- Check registry replication: `journalctl -u ic-registry-replicator`
- Verify network connectivity to NNS
- Confirm node is assigned to subnet in registry
- Check for key registration errors

**High Resource Usage**
- Review state size: Check `/var/lib/ic/data/`
- Monitor process metrics
- Check for memory leaks: `node_memory_*` metrics
- Verify no unexpected processes consuming resources

**Upgrade Failures**
- Check `orchestrator_replica_version_upgrade_prevented_total` metric
- Review upgrade logs for error messages
- Verify download URLs are accessible
- Confirm cryptographic hash verification passes

## Governance Participation

### Network Nervous System (NNS)

Node providers interact with the NNS for:

- **Node Registration**: Adding new nodes to the network
- **Subnet Assignment**: Nodes are assigned to subnets via NNS proposals
- **Upgrade Approval**: Replica and OS upgrades are approved through governance
- **Rewards**: Node provider rewards are managed by NNS

**Key NNS Concepts:**

- **Proposals**: Changes to the network require NNS proposals
- **Neurons**: ICP tokens locked for voting power
- **Voting**: Participate in governance decisions
- **Following**: Delegate voting to other neurons

For more information:
- [Understanding the Internet Computer's Network Nervous System](https://medium.com/dfinity/understanding-the-internet-computers-network-nervous-system-neurons-and-icp-utility-tokens-730dab65cae8)
- [List of NNS Proposals](https://github.com/dfinity/nns-proposals)

### Node Provider Proposals

Common proposal types for node providers:

1. **Add Node**: Register a new node
2. **Remove Node**: Decommission a node
3. **Update Node**: Modify node configuration
4. **Subnet Membership**: Assign nodes to subnets
