## Overview

Effective monitoring is critical for operating healthy Internet Computer nodes. The IC platform exposes comprehensive metrics through Prometheus-compatible endpoints, providing visibility into node health, performance, and network participation.

## Metrics Endpoints

### Replica Metrics (Port 9090)

The IC replica exposes comprehensive metrics about protocol operations:

**Endpoint Configuration** (from `rs/http_endpoints/metrics/src/lib.rs:126-146`):

```rust
// Default metrics address: [::]:9090
// Configuration:
let mut addr = "[::]:9090".parse::<SocketAddr>().unwrap();
addr.set_port(address.port());

// Service configuration:
.timeout(Duration::from_secs(config.request_timeout_seconds))
.layer(GlobalConcurrencyLimitLayer::new(
    config.max_concurrent_requests,
))
```

**Access the endpoint:**

```bash
curl http://[::1]:9090/metrics
# Or IPv4:
curl http://localhost:9090/metrics
```

<Info>
Metrics scraping has built-in timeout handling. The default timeout is configurable via `request_timeout_seconds`, and Prometheus scrape timeout is respected via the `X-Prometheus-Scrape-Timeout-Seconds` header.
</Info>

**Metric Collection:**

```rust
// From rs/http_endpoints/metrics/src/lib.rs:90-108
```

**Key Rotation Monitoring:**

```rust
// From rs/orchestrator/src/metrics.rs:119-135
pub fn observe_key_rotation_status(&self, status: KeyRotationStatus) {
    // Set current status and clear all other states
    // Don't clear error status when in transient states
    KeyRotationStatus::iter()
        .filter(|s| !status.is_transient() || !s.is_error())
        .for_each(|s| {
            self.key_rotation_status
                .with_label_values(&[s.as_ref()])
                .set((s == status) as i64);
        });
}
```

### Custom GuestOS Metrics

IC-specific metrics from GuestOS environment.

**Metric Generation** (from `ic-os/components/monitoring/guestos/custom-metrics.sh`):

```bash
#!/bin/bash
source /opt/ic/bin/logging.sh
source /opt/ic/bin/metrics.sh
source /opt/ic/bin/config.sh

# Version tracking
GUESTOS_VERSION_FILE="/opt/ic/share/version.txt"
if [ -r ${GUESTOS_VERSION_FILE} ]; then
    GUESTOS_VERSION=$(cat ${GUESTOS_VERSION_FILE})
    GUESTOS_VERSION_OK=1
else
    GUESTOS_VERSION="unknown"
    GUESTOS_VERSION_OK=0
fi

write_metric_attr "guestos_version" \
    "{version=\"${GUESTOS_VERSION}\"}" \
    "${GUESTOS_VERSION_OK}" \
    "GuestOS version string" \
    "gauge"
```

**Custom Metrics:**

```prometheus
# GuestOS version
guestos_version{version="<version>"}
  # Current GuestOS version (1=readable, 0=unknown)

# Boot status
guestos_boot_action{successful_boot="true"}
  # GuestOS boot status

# Configuration
guestos_config_version{version="<version>"}
  # GuestOS configuration version

# TEE (Trusted Execution Environment)
guestos_tee_configured
  # Whether VM is configured for TEE (1=yes, 0=no)
  
guestos_tee_active
  # Whether VM is running in TEE (1=yes, 0=no)
```

**TEE Metrics Implementation:**

```bash
# From custom-metrics.sh:49-76
function update_tee_metrics() {
    local metric_family="guestos_tee"
    clear_metrics "$metric_family"
    
    tee_configuration_str=$(get_config_value '.icos_settings.enable_trusted_execution_environment')
    
    if [[ "${tee_configuration_str}" == "true" ]]; then
        tee_configured=1
    else
        tee_configured=0
    fi
    
    write_metric_header "$metric_family" \
        "guestos_tee_configured" \
        "Indicates whether VM is configured to run in TEE" \
        "gauge"
    append_metric "$metric_family" "guestos_tee_configured" "" "${tee_configured}"
    
    write_metric_header "$metric_family" \
        "guestos_tee_active" \
        "Indicates whether VM is running in TEE" \
        "gauge"
    append_metric "$metric_family" "guestos_tee_active" "" "$SEV_ACTIVE"
}
```

## Metrics Collection with Prometheus

### Prometheus Configuration

Configure Prometheus to scrape IC node metrics:

```yaml
# prometheus.yml
global:
  scrape_interval: 15s
  evaluation_interval: 15s
  scrape_timeout: 10s

scrape_configs:
  # Replica metrics
  - job_name: 'ic-replica'
    static_configs:
      - targets: ['node1.example.com:9090', 'node2.example.com:9090']
        labels:
          cluster: 'ic-production'
          
  # Orchestrator metrics
  - job_name: 'ic-orchestrator'
    static_configs:
      - targets: ['node1.example.com:9091', 'node2.example.com:9091']
        labels:
          cluster: 'ic-production'
          
  # Node exporter metrics
  - job_name: 'node-exporter'
    static_configs:
      - targets: ['node1.example.com:9100', 'node2.example.com:9100']
        labels:
          cluster: 'ic-production'
```

### Service Discovery

For dynamic node discovery, use Prometheus service discovery:

```yaml
# Using file-based service discovery
scrape_configs:
  - job_name: 'ic-nodes'
    file_sd_configs:
      - files:
        - '/etc/prometheus/targets/ic-nodes-*.json'
        refresh_interval: 60s
```

**Target file format:**

```json
[
  {
    "targets": ["node1.example.com:9090"],
    "labels": {
      "job": "ic-replica",
      "environment": "production",
      "node_id": "<node_id>"
    }
  }
]
```

## Alerting Rules

### Prometheus Alerting Configuration

```yaml
# /etc/prometheus/rules/ic-alerts.yml
groups:
  - name: ic_critical
    interval: 30s
    rules:
      # Node offline
      - alert: ICNodeDown
        expr: up{job=~"ic-.*"} == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "IC Node {{ $labels.instance }} is down"
          description: "Node has been unreachable for 5 minutes"

      # Critical errors
      - alert: ICCriticalError
        expr: rate(orchestrator_cup_deserialization_failed_total[5m]) > 0
        labels:
          severity: critical
        annotations:
          summary: "Critical CUP deserialization error on {{ $labels.instance }}"
          description: "Investigate immediately - check orchestrator logs"

      # Key rotation failure
      - alert: ICKeyRotationError
        expr: orchestrator_key_rotation_status{status="Error"} == 1
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "Key rotation error on {{ $labels.instance }}"
          description: "Check key rotation logs and registry connectivity"

  - name: ic_warnings
    interval: 60s
    rules:
      # High CPU
      - alert: ICHighCPU
        expr: (100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)) > 80
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "High CPU usage on {{ $labels.instance }}"
          description: "CPU usage is {{ $value }}%"

      # High memory
      - alert: ICHighMemory
        expr: (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 90
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage on {{ $labels.instance }}"
          description: "Memory usage is {{ $value }}%"

      # Disk space
      - alert: ICLowDiskSpace
        expr: (1 - (node_filesystem_avail_bytes{mountpoint="/var/lib/ic"} / node_filesystem_size_bytes{mountpoint="/var/lib/ic"})) * 100 > 80
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Low disk space on {{ $labels.instance }}"
          description: "Disk usage is {{ $value }}%"

      # Registry sync lag
      - alert: ICRegistryOutdated
        expr: time() - firewall_registry_version > 7200
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "Registry synchronization lag on {{ $labels.instance }}"
          description: "Registry hasn't updated in {{ $value }} seconds"

      # Upgrade issues
      - alert: ICUpgradeCheckFailures
        expr: rate(orchestrator_failed_consecutive_upgrade_checks_total[15m]) > 0
        for: 30m
        labels:
          severity: warning
        annotations:
          summary: "Consecutive upgrade check failures on {{ $labels.instance }}"
          description: "Upgrade checks are failing - check network and registry"
```


## Monitoring Best Practices

<Steps>
  <Step title="Monitor All Endpoints">
    Scrape all three metrics endpoints (replica, orchestrator, node_exporter) to get complete visibility.
  </Step>

  <Step title="Set Appropriate Scrape Intervals">
    - High-frequency metrics (replica): 15s
    - Medium-frequency metrics (orchestrator): 30s
    - Low-frequency metrics (node_exporter): 60s
  </Step>

  <Step title="Configure Critical Alerts">
    Alert on critical errors immediately - they indicate serious issues requiring investigation.
  </Step>

  <Step title="Track Trends">
    Monitor long-term trends for:
    - Resource usage growth
    - Error rates
    - Performance degradation
  </Step>

  <Step title="Document Baseline">
    Establish baseline metrics for normal operation to identify anomalies.
  </Step>

  <Step title="Test Alert Paths">
    Regularly test alert delivery to ensure notifications reach the right people.
  </Step>
</Steps>

## Troubleshooting Monitoring Issues

### Metrics Not Available

**Check service status:**
```bash
systemctl status ic-replica
systemctl status ic-orchestrator
systemctl status node_exporter
```

**Test endpoint directly:**
```bash
curl -v http://localhost:9090/metrics
curl -v http://localhost:9091/metrics
curl -v http://localhost:9100/metrics
```

**Check firewall rules:**
```bash
# Firewall is managed by orchestrator
# Check current rules:
iptables -L -n -v
```

### High Scrape Duration

<Note>
Adapter metrics collection has built-in timeout protection to prevent slow adapters from blocking replica metrics.
</Note>

```rust
// From rs/http_endpoints/metrics/src/lib.rs:24-28
const DEFAULT_ADAPTER_COLLECTION_TIMEOUT: Duration = Duration::from_secs(1);
const PROMETHEUS_TIMEOUT_FRACTION: f64 = 0.5;
const PROMETHEUS_TIMEOUT_HEADER: &str = "X-Prometheus-Scrape-Timeout-Seconds";
```

If scrapes are timing out:
- Increase Prometheus scrape timeout
- Check for performance issues on the node
- Review adapter health if replica metrics are slow

### Missing Custom Metrics

**Check metric scripts:**
```bash
# Verify custom metric generation
systemctl status metrics_tool.service
systemctl status custom-metrics.service

# Check timers
systemctl list-timers | grep metrics
```

**Manually run metric collection:**
```bash
/opt/ic/bin/custom-metrics.sh
```
