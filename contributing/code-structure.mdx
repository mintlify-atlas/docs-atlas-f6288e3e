The Internet Computer repository is organized as a monorepo containing all components needed to build and run the IC protocol implementation.

## Repository Overview

The repository contains several major directories, each serving a specific purpose:

```
ic/
├── rs/              # Rust implementation of the replica and core services
├── ic-os/           # Operating system images (SetupOS, HostOS, GuestOS)
├── packages/        # Standalone packages and utilities
├── ci/              # Continuous integration scripts and configuration
├── bazel/           # Bazel build system configuration
└── third_party/     # Third-party dependencies
```

## Core Directories

### ic-os/ - Operating Systems

IC-OS contains the operating system images that run on Internet Computer nodes. These are minimal, security-hardened systems.

<CodeGroup>
```text Structure
ic-os/
├── setupos/      # Initial boot and installation OS
├── hostos/       # Host operating system
├── guestos/      # Guest OS running the replica
├── components/   # Shared components across all OS variants
└── bootloader/   # Boot loader configuration
```
</CodeGroup>

<CardGroup cols={3}>
  <Card title="SetupOS" icon="circle-play">
    Responsible for booting a new replica node and installing HostOS and GuestOS.
  </Card>
  
  <Card title="HostOS" icon="server">
    Runs on the host machine. Launches and manages the GuestOS virtual machine. Intentionally limited in capabilities.
  </Card>
  
  <Card title="GuestOS" icon="microchip">
    Runs inside a VM on HostOS. Executes the core IC protocol and replica software.
  </Card>
</CardGroup>

<Info>
IC-OS images are built as Docker images first, then transformed into bootable bare-metal or virtual-metal images. See the [IC-OS README](https://github.com/dfinity/ic/blob/master/ic-os/README.adoc) for details.
</Info>

#### Image Variants

Each OS has multiple build targets:

- **prod** - Production images (no console access)
- **dev** - Development images (console access enabled, username/password: `root`/`root`)
- **dev-malicious** (GuestOS only) - For testing malicious behavior
- **recovery** (GuestOS only) - For subnet recovery

### packages/ - Standalone Packages

The `packages/` directory contains standalone Rust packages that can be used independently:

```text
packages/
├── canlog/                        # Canister logging utilities
├── ic-ed25519/                    # Ed25519 signature implementation
├── ic-ethereum-types/             # Ethereum type definitions
├── ic-secp256k1/                  # secp256k1 implementation
├── ic-secp256r1/                  # secp256r1 implementation
├── ic-signature-verification/     # Signature verification utilities
├── icrc-ledger-types/             # ICRC ledger type definitions
└── pocket-ic/                     # PocketIC local testing framework
```

These packages are often published to crates.io for use by external developers.

### ci/ - Continuous Integration

Contains all CI/CD infrastructure:

```text
ci/
├── container/       # Docker container definitions for builds
├── scripts/         # Build and test automation scripts
├── actions/         # GitHub Actions workflows
└── src/             # CI tooling source code
```

Key scripts:

- `ci/container/container-run.sh` - Enter dev container
- `ci/container/build-ic.sh` - Build IC components
- `ci/scripts/rust-lint.sh` - Rust linting

## Build System

### Bazel

The project uses [Bazel](https://bazel.build) as its primary build system:

- `BUILD.bazel` files throughout the repository define build targets
- `WORKSPACE.bazel` defines workspace-level configuration
- `MODULE.bazel` defines external dependencies (Bzlmod)

### Cargo

Rust packages also use Cargo for dependency management:

- Root `Cargo.toml` defines workspace-level dependencies
- Individual crates have their own `Cargo.toml` files
- The workspace uses Cargo workspace inheritance for dependency version management

<Note>
When adding dependencies, add them to the `[workspace.dependencies]` section in the root `Cargo.toml`, then reference them with `{ workspace = true }` in individual crate manifests.
</Note>

## Common Patterns

### Workspace Members

The root `Cargo.toml` lists all workspace members (534 packages as of this writing). This ensures:

- Unified dependency versions across the workspace
- Single `Cargo.lock` file
- Faster builds through shared dependencies

### Protobuf Generation

Many packages have a `protobuf_generator` subdirectory:

```text
rs/crypto/
├── src/              # Main crypto implementation
└── protobuf_generator/  # Generates Rust code from .proto files
```

### Test Utilities

Most major subsystems have associated test utilities:

```text
rs/crypto/
├── src/              # Production code
└── test_utils/       # Testing helpers and mocks
```
