# Inter-Canister Calls

Inter-canister calls enable canisters to communicate with each other, forming the foundation of composable applications on the Internet Computer.

## Call Model

The IC uses an asynchronous message-passing model for inter-canister communication:

1. **Caller initiates** - Sends a request to the callee
2. **Request queued** - Message enters callee's input queue
3. **Callee processes** - Executes the method when scheduled
4. **Response sent** - Callee replies or rejects
5. **Callback executes** - Caller processes the response

<Note>
Inter-canister calls are fully asynchronous. The caller doesn't block waiting for a response.
</Note>

## Making Calls

### Basic Call Flow

From `rs/rust_canisters/dfn_core/src/api.rs:304-338`:

```rust
pub fn call_raw(
    id: CanisterId,
    method: &str,
    data: &[u8],
    on_reply: fn(ptr: *mut ()),
    on_reject: fn(ptr: *mut ()),
    on_cleanup: Option<fn(ptr: *mut ())>,
    env: *mut (),
    funds: Funds,
) -> i32 {
    unsafe {
        ic0::call_new(
            id.get().as_slice().as_ptr() as u32,
            id.get().as_slice().len() as u32,
            method.as_ptr() as u32,
            method.len() as u32,
            on_reply as usize,
            env as u32,
            on_reject as usize,
            env as u32,
        );
        ic0::call_data_append(data.as_ptr() as u32, data.len() as u32);
        if funds.cycles > 0 {
            call_cycles_add(funds.cycles);
        }
        if let Some(on_cleanup) = on_cleanup {
            ic0::call_on_cleanup(on_cleanup as usize, env as u32);
        }
        ic0::call_perform()
    }
}
```

### Call Phases

#### 1. Setup Phase

Configure the call using `ic0.call_new`:
- Specify target canister ID
- Specify method name
- Register reply and reject callbacks

#### 2. Data Phase

Build the request:
- Add payload with `ic0.call_data_append`
- Attach cycles with `ic0.call_cycles_add`
- Register cleanup with `ic0.call_on_cleanup`

#### 3. Execution Phase

Send the call:
- Call `ic0.call_perform`
- Check return code (0 = success)

### Using Futures

Higher-level API using async/await:

```rust
pub async fn call_bytes(
    id: CanisterId,
    method: &str,
    data: &[u8],
    funds: Funds,
) -> impl Future<Output = FutureResult<Vec<u8>>> {
    fn callback(future_ptr: *mut ()) {
        let ref_counted = unsafe {
            RefCounted::from_raw(future_ptr as *const RefCell<CallFuture>)
        };
        let mut future = ref_counted.borrow_mut();
        future.result = Some(match reject_code() {
            0 => Ok(arg_data()),
            n => Err((Some(n), reject_message())),
        });
        if let Some(waker) = future.waker.take() {
            waker.wake();
        }
    }
    // ... setup and perform call
}
```

## Response Handling

### Success Response

When the callee calls `ic0.msg_reply`:

1. Reply callback executes
2. Response data available via `ic0.msg_arg_data_copy`
3. Refunded cycles available via `ic0.msg_cycles_refunded`

### Reject Response

When the callee calls `ic0.msg_reject` or traps:

1. Reject callback executes
2. Reject code available via `ic0.msg_reject_code`
3. Reject message available via `ic0.msg_reject_msg_copy`
4. Cycles refunded automatically

### Cleanup Callback

Executed when a call is cancelled before completion:

- Called if the calling canister is stopped/deleted
- Called if the call times out
- Allows resource cleanup
- Does not receive response data

## Cycles in Calls

### Sending Cycles

Attach cycles before calling `ic0.call_perform`:

```rust
unsafe {
    ic0::call_cycles_add128(amount_high, amount_low);
}
```

<Warning>
Cycles are deducted from the caller's balance when the call is sent, not when it completes.
</Warning>

### Accepting Cycles

The callee can accept cycles:

```rust
let available = msg_cycles_available128();
let accepted = msg_cycles_accept128(amount_high, amount_low);
```

Unaccepted cycles are refunded to the caller.

### Refunds

Cycles are refunded when:
- Callee doesn't accept all cycles
- Call is rejected
- Callee traps

Access refunded cycles in callbacks:

```rust
let refunded = msg_cycles_refunded128();
```

## Best-Effort Calls

For less critical calls that can tolerate failures:

```rust
unsafe {
    ic0::call_with_best_effort_response(timeout_seconds);
}
```

Benefits:
- Lower latency (no consensus required)
- Cheaper (fewer resources)
- Timeout protection

Limitations:
- May not complete if subnet is busy
- No guaranteed delivery
- Response not certified

## Guaranteed Response Calls

Default call type with strong guarantees:

- Response is guaranteed (or reject)
- Goes through consensus
- State changes are committed
- Higher resource cost

Use for:
- Critical operations
- State modifications
- Operations requiring consensus

## Call Patterns

### Request-Response

Simple one-way call:

```rust
let result = call_bytes(
    target_canister,
    "method_name",
    &encode_args(),
    Funds::zero()
).await?;
```

### Fan-Out

Call multiple canisters in parallel:

```rust
let futures = canister_ids.iter().map(|id| {
    call_bytes(*id, "fetch_data", &[], Funds::zero())
});

let results = futures::future::join_all(futures).await;
```

### Call Chain

Sequential calls:

```rust
let data1 = call_bytes(canister1, "get_info", &[], Funds::zero()).await?;
let data2 = call_bytes(canister2, "process", &data1, Funds::zero()).await?;
let result = call_bytes(canister3, "finalize", &data2, Funds::zero()).await?;
```

### Callback Composition

Nested callbacks for complex flows:

```rust
call_with_callbacks(
    canister_a,
    "method_a",
    &data,
    || {
        // On success, make another call
        call_with_callbacks(
            canister_b,
            "method_b",
            &response_data(),
            || { /* final success */ },
            || { /* final reject */ }
        );
    },
    || { /* first call rejected */ }
);
```

## Error Handling

### Reject Codes

From the Internet Computer specification:

- `0` - No error (success)
- `1` - SYS_FATAL - Fatal system error
- `2` - SYS_TRANSIENT - Transient system error
- `3` - DESTINATION_INVALID - Invalid destination
- `4` - CANISTER_REJECT - Canister explicitly rejected
- `5` - CANISTER_ERROR - Canister trapped or errored

### Handling Rejects

```rust
match reject_code() {
    0 => {
        // Success case
        let data = arg_data();
        process_response(data);
    },
    2 => {
        // Transient error - retry may succeed
        schedule_retry();
    },
    4 | 5 => {
        // Canister reject/error - check message
        let msg = reject_message();
        handle_canister_error(msg);
    },
    _ => {
        // Other errors
        handle_system_error();
    }
}
```

## Limits and Constraints

### Message Size

From the IC specification:

- **Request payload** - Up to 2 MB
- **Response payload** - Up to 2 MB
- **Chunked uploads** - For larger data

### Queue Capacity

From `rs/replicated_state/src/canister_state/queues/mod.rs`:

```rust
pub const DEFAULT_QUEUE_CAPACITY: usize = 500;
```

Per-queue limits:
- 500 messages per input queue (default)
- Configurable based on subnet capacity
- Backpressure when queues fill

### Callback Limits

Subnet-wide limits:
- Soft cap on outstanding callbacks
- Prevents resource exhaustion
- Calls may be rejected if exceeded

### Cycles Limits

Maximum cycles per call:
- Limited by canister's balance
- Limited by u128 max value
- Execution cost deducted

## Best Practices

### Design Patterns

1. **Idempotency** - Make calls safe to retry
2. **Timeouts** - Use deadlines for time-sensitive operations
3. **Error handling** - Always handle both success and reject
4. **Resource limits** - Stay within queue and callback limits

### Performance

1. **Batch calls** - Combine multiple operations when possible
2. **Parallel execution** - Use fan-out for independent operations
3. **Avoid deep chains** - Long call chains increase latency
4. **Cache results** - Reduce redundant cross-canister calls

### Security

1. **Validate caller** - Check `ic0.msg_caller()` for authorization
2. **Validate cycles** - Check attached cycles before expensive work
3. **Rate limiting** - Protect against call flooding
4. **Input validation** - Sanitize all incoming data

### Reliability

1. **Handle transient errors** - Retry with exponential backoff
2. **Circuit breakers** - Fail fast when a callee is consistently down
3. **Cleanup handlers** - Release resources if calls are cancelled
4. **Monitor queues** - Track queue depths to detect issues

## Common Pitfalls

### Double Reply

Calling `ic0.msg_reply()` or `ic0.msg_reject()` twice causes a trap:

```rust
// BAD - will trap on second reply
if condition {
    msg_reply();
}
msg_reply(); // Trap!
```

### Forgetting to Reply

Not calling reply or reject leaves the caller waiting:

```rust
// BAD - no response sent
fn my_method() {
    process_data();
    // Forgot to call msg_reply()!
}
```

### Callback Leaks

Not cleaning up resources in reject/cleanup callbacks:

```rust
// BAD - memory leak if call is rejected
let data = allocate_large_buffer();
call_bytes(canister, "method", &[], Funds::zero()).await?;
// If rejected, data is never freed
```

### Insufficient Cycles

Not checking if enough cycles are available:

```rust
// BAD - may fail if balance is low
call_cycles_add(large_amount);
call_perform(); // May fail silently
```
